# Spec: 도쿄 지역 가이드 — 지도 & 인포그래픽 시스템

## Overview

GSF-Blog 도쿄 지역 가이드 시리즈(E0~E11)에 고급 지도와 데이터 인포그래픽을 체계적으로 삽입하기 위한 기술 + 디자인 통합 스펙.
**"먼저 독립 시안(HTML)으로 완성 → 검증 후 블로그에 통합"** 2단계 워크플로를 채택한다.

---

## 실패 이력 요약 (Root Cause)

### 3회 연속 실패의 근본 원인

| # | 시도 | 실패 원인 |
|---|------|-----------|
| 1 | SVG 인라인을 Markdown(.md)에 직접 삽입 | Astro의 rehype 파이프라인이 `<defs>`, `<linearGradient>`, `filter`, `stroke-dasharray`, `fill-opacity` 등 SVG 고급 속성을 **새니타이징(strip)** |
| 2 | 색상 팔레트 수정 시도 | 배경(`#0f1729`) vs fill(`#1e3a8a`) 대비 비율 1.2~1.5:1 → WCAG AA(4.5:1) 미달. 새니타이징 해결해도 안 보임 |
| 3 | 구 경계 path 정확도 부족 | GeoJSON 기반이라 주석 달았으나, 실제로는 에이전트 추정 좌표. 지리적 정확성 제로 |

### 핵심 교훈

> **기획 오류가 아니라 "플랫폼(Astro) 제약 미검증"이 근본 원인.**
> SVG를 .md 파일에 인라인 삽입하면 Astro rehype가 핵심 속성을 제거한다.
> → 구현 전에 반드시 **독립 HTML로 시각 검증** 후 통합해야 한다.

- E0/E1에 프리미엄급 지도 + 데이터 인포그래픽 삽입

- 라이트/다크 모드 양립

| 요소 | A안 (다크+지도) | B안 (다크+카드) | C안 (라이트+지도) | D안 (블루+지도) |

**채택 방향: B안 + D안 하이브리드**

- 대비 비율 WCAG AA 보장을 위해 fill 색상 대폭 밝게 조정

## 구현 전략 — 2단계 워크플로

### Phase 1: 독립 시안 제작 + 시각 검증

```
GSF-Blog/prototypes/
├── e0-ward-grouping-map.html
├── e1-core3-infographic.html

```

- **독립 HTML 파일**로 SVG/CSS를 100% 자유롭게 작성
- 브라우저에서 직접 열어 시각 검증 (Astro 새니타이징 무관)
- 사용자 승인 후 Phase 2로 이행

### Phase 2: Astro 컴포넌트 통합 (권장 옵션 A)

| 옵션 | 방식 | 장점 | 단점 | 권장 |
|------|------|------|------|------|
| **A. Astro 컴포넌트** | `.astro` 파일, slug 조건부 렌더링 | SVG 100% 보존, hover 가능 | slug 분기 필요 | ⭐⭐⭐ |
| B. 이미지화 | HTML→WebP, `![](...)` 삽입 | 가장 단순 | 인터랙션 불가 | ⭐⭐ |
| C. rehype 커스텀 | `rehype-sanitize` 스키마 확장 | .md 유지 | 보안 리스크 | ⭐ |

## E0 — 23구 브랜드 그루핑 지도 설계

- 23구 **반추상 특징적 윤곽** (주요 형태 보존, 15~25KB)
- 8개 에피소드 그룹별 컬러 코딩
- 범례, 방위, 출처 필수
- CSS-only hover (JS 제로)

## E1 — 핵심 3구 데이터 비교 설계

**4개 구성 요소:**
1. **위치 지도**: 23구 아웃라인 + 3구 하이라이트 + 랜드마크 핀
2. **KPI 카드 4열**: 맨션 ㎡단가, 1LDK 월세, 외국인 비율, 세대 연소득
3. **비교 막대 차트**: 3구 가로 비교
4. **도넛 차트**: 외국인 구성비 또는 주거 타입 분포

## Phase 1 프로토타입 리뷰 결과 (2026-05-12)

### 사용자 피드백

1. **지도 추상화 과잉**: "실제 땅의 모양이 반영되면 좋겠다" → 수동 좌표 path를 GeoJSON 기반 실좌표로 교체
2. **색상 과다**: "그래프가 너무 알록달록" → 8색 무지개 → 2톤+Accent 절제 팔레트로 전환

### 전문가 리뷰 점수

| 관점 | 점수 | 핵심 병목 |
|------|------|----------|
| 디자인 | 55/100 | 지도 추상화 + 색상 8색 과다 |
| 시스템 | 72/100 | 수동 좌표 SVG, 확장성 낮음 |
| 비즈니스 | 60/100 | 데이터 스토리텔링 부재 |

### 확정된 개선 방향

**P0 — 지도 GeoJSON 전환:**
- 국토수치정보(MLIT) Shapefile → Mapshaper 85% simplify → SVG path
- 강(荒川·隅田川) 2~3개 + 동경만 해안선 리얼하게
- 목표: 실제 행정구역 형태 보존, 15~25KB 이내

**P0 — 색상 2톤 절제 전략:**
```css
--ig-highlight: #3b82f6;        /* Blue — 현재 에피소드 하이라이트 */
--ig-neutral-fill: rgba(100,116,139,0.15);  /* Slate — 나머지 구 */
--ig-accent: #f59e0b;           /* Gold — 최고값/인사이트 포인트 */
```
- E0 전체 지도: 모든 에피소드를 흐린 Slate로, 호버시에만 해당 에피소드 색상 활성화
- E1~E11 개별 지도: 해당 구만 Blue 하이라이트, 나머지 dim

**P1 — 데이터 스토리텔링:**
- KPI 카드에 원화(₩) 환산 병기
- 각 차트 하단에 인사이트 한 줄 ("→ 23구 평균의 2.1배")
- 서울 비교 기준 추가 (강남/마포 등)

## 자동화 아이디어 (2026-05-12 기록)

### A. `gen_map.py` 통합 CLI

- **트리거**: E2~E11 새 에피소드 작성 시
- **입력**: `python3 gen_map.py --ep 2 --wards 新宿区,渋谷区,文京区`
- **출력**: 해당 에피소드 하이라이트 지도 SVG + HTML 자동 생성
- **현재 갭**: `EP_GROUPS` 딕셔너리를 gen_e0.py 내에서 수동 수정해야 함
- **ROI**: 12편 × 지도 1개 = 12회 반복 작업 자동화

### B. Mapshaper 크기 자동 튜닝 스크립트

- **트리거**: LCP 예산(SVG 30KB) 초과 감지 시
- **입력**: GeoJSON 파일 + 목표 KB
- **출력**: 이진 탐색으로 목표 크기에 맞는 simplify% 자동 산출
- **현재 갭**: simplify 3% 고정. 구 경계 복잡도에 따라 KB가 달라짐
- **구현 난이도**: 낮음 (mapshaper CLI + Python subprocess + bisect)
