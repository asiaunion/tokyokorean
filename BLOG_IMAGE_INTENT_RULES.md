# [GSF-Blog · AG 필수 지식] 여행·산책 포스트 이미지 작업 — 사용자 의도 해석 및 실행 규칙

> **적용 범위:** Track 1 수동·하이브리드 콘텐츠, 특히 `ginza-weekend-walking-guide` 유형(체험담 + 장소별 섹션 + 히어로/본문 2장).  
> **목표:** 사용자가 채팅에 사진을 붙이거나 “이 정도면 충분해”라고 말했을 때, AG가 **추측으로 Downloads를 뒤지지 않고** 의도에 맞게 끝까지 맞추는 것.

## 0. AG가 먼저 이해할 “사용자 의도”의 본질

사용자가 이미지 수정을 요청할 때, 표면 요청 뒤에 보통 다음 **4가지**가 있습니다.

| 깊은 의도 | 표면 요청 예시 | AG가 해야 할 것 |
|-----------|----------------|-----------------|
| **신뢰** | “히어로가 이상해”, “부부 사진 말고” | 공개 여행 가이드에 맞는 **주제 장면**만 히어로에 사용 |
| **가독성** | “같은 사진이 두 번”, “모자이크 너무해” | hero ≠ body, **건물은 선명·인물만 완화** |
| **프라이버시** | “얼굴 블러”, “표정 문장 이상해” | 식별 불가 수준 + **본문·캡션을 편집 방식과 일치** |
| **통제** | “내가 고른 사진으로”, “Cursor가 했어” | **사용자·채팅 첨부·지정 폴더**만 원본으로 사용, 임의 파일 금지 |

**한 줄 원칙 (반복 암기):**  
> 여행 가이드 이미지는 “사용자가 고른 2장(히어로=그 글의 얼굴, 본문=해당 절의 증거)”을 **가볍게** 다듬는 것이지, AG가 임의로 사진을 골라 **무겁게** 가리는 작업이 아니다.

## 1. 실패 사례 — `ginza-weekend-walking-guide` (절대 재현 금지)

### 커밋 `626a226` 등에서 한 일 (전부 금지)

1. **`Downloads/도쿄타워 배경.JPG`** 류를 히어로로 사용 → 실내 부부 셀카 + 도쿄타워. **§1 츠타야·주말 긴자 산책과 무관.**
2. **hero.webp = 1.webp 동일 파일** → 페이지 상단 + 본문 §3에 **동일 이미지 2회**.
3. 보행자 천국에 **넓은 픽셀 모자이크** → 야마노·미츠코시·거리 분위기 파괴.
4. 캡션만 “편집/모자이크”로 바꾸고 본문에 **「표정」「사진 왼쪽」** 유지 → 글과 사진 모순.
5. “privacy safe” 보고했으나 사용자 기준 **실패** → **보고 전에 체크리스트(§7) 필수.**

### 정답 방향 (커밋 `d9e2d26`, `a7066f6`)

| 역할 | 내용 | 프라이버시 |
|------|------|------------|
| **hero.webp** | 츠타야서점 간판(蔦屋書店) — §1과 일치 | 인물 없음 |
| **1.webp** | 보행자 천국, **왼편 야마노·오른편 미츠코시** — §3과 일치 | 하단 인물 **약한 가우시안 블러**, **실루엣 유지** |

사용자 최종 피드백: **“이 정도면 충분해”** = 블러는 **약하게**, 사람 **형태는 보이게**.

## 2. 요청 해석 — 사용자 말 → AG 행동 매핑

요청이 애매할 때 아래표로 **의도를 먼저 확정**한 뒤 작업한다.

| 사용자 표현 | 의미 (추론) | AG 행동 |
|-------------|-------------|---------|
| “히어로가 이상해 / 부부 사진” | 히어로 주제·톤 불일치 | hero 교체. **개인 셀카·실내·타 지역 랜드마크** 제외 |
| “같은 사진 두 번” | hero = body 중복 | **서로 다른 원본 2장**. MD5 다름 확인 |
| “모자이크 / 너무 가려” | 편집 과함 | **모자이크 금지**. 가우시안 블러 + 강도 **낮춤** |
| “사람 형태는 보이게” | 인파·분위기는 살림 | 블러 **하단·얼굴 구간만**, radius/강도 **약하게** |
| “야마노 왼쪽” | 본문 §3 지리 정합 | body 컷에 **山野楽器 왼편** 필수. 크롭 시 간판 잘리지 않게 |
| “채팅에 사진 붙였어 / 골라서 해줘” | 원본·선정권은 사용자 | **첨부·지정 폴더만** 사용. Downloads 자동 탐색 **금지** |
| “이 정도면 충분해” | 현재 편집 강도 **상한** | 더 세게 블러·모자이크 **하지 말 것** |
| “커밋하고 배포” | 게시까지 원함 | 커밋·push (사용자 지시 시만). 그 전엔 **로컬 변경 + 보고** |
| “AG에게 학습시켜” | Knowledge·런북 갱신 | 이 문서 반영. **626a226 패턴 재사용 금지** 명시 |

**불명확할 때 (질문 1~2개만, 과하지 않게):**
- “히어로는 §1 츠타야/정원 중 어느 カット로 할까요?”
- “본문 블러는 지금보다 더 약하게 / 조금만 더 강하게 중 어느 쪽인가요?”

**질문하지 말아야 할 때:** 사용자가 이미 **10장 첨부 + ‘골라서 수정’** → 표로 후보 제시 후 **합리적 기본안(아래 §3)으로 진행**하고, 선택 근거를 보고서에 적는다.

## 3. 표준 아키텍처 — “Option A” (2장 구조)

### 3-1. 파일·역할 (slug 예: `ginza-weekend-walking-guide`)

| | hero | body (`-1.webp`) |
|---|------|------------------|
| **몇 번 나오나** | og + 페이지 상단 1회 | 마크다운 본문 1회 |
| **주제** | 글의 **첫 인상** = 보통 §1 장소 | **해당 절의 증거** = 예: §3 보행자 천국 |
| **인물** | 없거나 식별 불가 | 있어도 됨 → **얼굴만 약한 블러**, 실루엣 유지 |
| **금지** | 본문과 동일 파일, 개인 셀카, 무관 랜드마크 | 전면 모자이크, 건물 가림, hero와 동일 |

**기술:** WebP, max width ~1200px, quality ~80–82.

### 3-2. 섹션 ↔ 이미지 매핑 (ginza-weekend 예시)

| 섹션 | 내용 | 이미지 |
|------|------|--------|
| §1 | 츠타야서점 GINZA SIX 6F | **hero** 후보 |
| §2 | GINZA SIX Garden 13F | hero **대안**(나무·전망, 인물 없음) |
| §3 | 보행자 천국, 야마노 지가 | **body `-1.webp` 전용** |
| 결론 | 시리즈 링크 등 | 이미지 추가 불필요(중복 방지) |

### 3-3. 사진 후보가 여러 장일 때 선정 로직

사용자가 N장 첨부했을 때 AG는 **작업 전** 아래 표를 채워 사용자에게 보여준다(또는 보고서에 포함).

| # | 설명 | hero | body | 제외 사유 |
|---|------|:----:|:----:|-----------|
| 1 | 츠타야 간판, 인물 없음 | ◎ | | §1·OG에 최적 |
| 2 | 보행자천국, 야마노 좌 | | ◎ | §3·지가 설명과 일치 |
| 3 | Ginza Place / 교협당 | | △ | body 대안(야마노 없으면) |

## 4. 본문 이미지 편집 — 프라이버시 “정답 스펙”

### 4-1. 하지 말 것

- 픽셀 **모자이크** (특히 건물·간판까지)
- 화면 **하반 전체**를 불투명 블러로 덮기
- “privacy safe”라고 **검증 없이** 보고

### 4-2. 해야 할 것 (사용자 확정 수준)

1. **가우시안 블러**만 사용.
2. **적용 구간:** 프레임 **아래쪽·인물 밀집 구간**만 점진적(그라데이션 마스크).
3. **강도:** 약함 — **인파 실루엣·옷차림·걸음**이 보일 것.
4. **건물:** 야마노(山野楽器), 미츠코시, 교분칸 등 **선명**.
5. **크롭:** 3:2 등 가로 비율 시 **야마노 간판이 잘리지 않게** (상단만 과도하게 자르지 말 것).

### 4-3. 강도 조절 대화

| 사용자 말 | AG 조치 |
|-----------|---------|
| “블러 더 약하게 / 형태 보이게” | radius↓, 합성 강도↓(예: 50%→38%), 구간을 **더 아래**로 |
| “이 정도면 충분해” | **현재 강도를 상한**으로 Knowledge에 기록, 추가 강화 금지 |
| “얼굴만” | 하단 중앙 인물 위주; **전체 폭 블러** 지양 |

### 4-4. 편집 후 반드시 할 일

- ko / en / ja **동시** 수정(§5).
- 로컬에서 hero·body **미리보기** 후 보고.

## 5. 본문·캡션 — 이미지와 문장 정합 (3언어)

이미지를 바꾸거나 블러하면 **마크다운만 고치지 말고** 체험담 문장도 맞춘다.

### 5-1. 제거·수정할 표현

| 제거 (블러·후면 촬영과 모순) | 대체 방향 |
|------------------------------|-----------|
| 표정, 얼굴, expression | 거리 분위기, 인파, 여유로운 리듬 |
| 사진을 보아도, 写真を見ても | (삭제 또는) 거리를 걸으며, 歩きながら |
| 사진 왼쪽, left of the photo | **거리 왼편**, left side of the **street** |
| 모자이크 (캡션) | **얼굴 블러 / softly blurred / ぼかし** |

### 5-2. 캡션 템플릿 (ginza §3)

**KO:**  
`*주말 토·일·공휴 낮 시간, 차량 통행이 금지된 츄오도리 보행자 천국. 왼편 건물 일대에 야마노 악기 긴자 본점이 있습니다. (프라이버시 보호를 위해 전경 인물 얼굴에 블러 처리를 했습니다.)*`

**EN:**  
`*(Foreground faces have been softly blurred to protect privacy.)*`

**JA:**  
`*(手前の人物の顔にはぼかし処理を施しています。)*`

### 5-3. 사실 수치 (이미지와 별도, 함께 검증)

- 야마노 긴자 본점: **㎡당 약 6,710만 엔 (2026)**, **20년 연속** 1위 (**21년 아님**).
- 보행자 천국·Garden 시간: `ginza.jp`, `ginza6.tokyo/hours`와 일치.
- `references ⊆ sources`, `content-depth-v2`, KO 한글 **1,800~2,300자**.

## 6. AG 작업 순서 (권장 파이프라인)

1. **의도 파악 및 사진 후보 분류**: 사용자가 제공하거나 첨부한 사진 리스트가 있을 때, 3-3의 '사진 후보 선정 로직'을 적용하여 1순위와 대안, 배제군(개인 셀카 등)을 매핑하고 선정의 타당성을 수립합니다.
2. **에셋 분리 및 가공 (Option A)**:
   - `hero.webp`: 본문 테마(§1 등)에 부합하는 정경(인물 배제)을 가로 ~1200px, WebP 포맷(quality ~80-82)으로 저장합니다.
   - `1.webp` (혹은 body 이미지): 야마노 악기 및 핵심 건물이 가려지지 않게 비율을 맞춰 크롭한 뒤, 하단 얼굴 영역에 약한 가우시안 블러를 적용해 실루엣은 완전히 보존하고 식별만 불가하게 처리합니다.
3. **3개 국어 마크다운 동시 갱신**: frontmatter의 `ogImage` 선언을 `webp` 확장자로 갱신하고, 본문에 이미지를 알맞게 복구시킵니다. 5-1에 명시된 묘사 모순 문구("표정", "사진 왼쪽" 등)를 자연스러운 묘사("거리 분위기", "거리 왼편")로 대체합니다.
4. **팩트 데이터 크로스 체크**: 2026년 공시지가 수치(㎡당 약 6,710만 엔, 20년 연속 전국 1위) 및 가이드 일정이 모든 번역 파일 및 인접 포스트와 빈틈없이 동기화되었는지 검증합니다.
5. **로컬 빌드 검증**: `pnpm build`를 수행하여 Pagefind 다국어 인덱싱과 정적 라우트 프리렌더링이 빌드 에러 없이 성공적으로 마무리되는지 1차 팩트 체크합니다.
6. **최종 승인 및 Vercel 배포**: 사용자의 배포 지시 전까지는 로컬에 반영한 뒤 승인을 구하며, 승인 및 지시가 떨어지는 즉시 `git push`를 진행해 원격 Vercel의 빌드가 성공(`● Ready`)하는지 3중 검증합니다.
