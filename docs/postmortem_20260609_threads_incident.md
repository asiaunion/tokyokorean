## 1. 사건 개요 및 타임라인 (Timeline)

* **오전 09:00 무렵**: 사용자가 "오늘 아침 9시 예약 포스팅 중, X만 정상이고 LinkedIn과 Threads에 히어로 이미지가 보이지 않는다. 영어 버전을 수정된 스킬로 지금 다시 포스팅해달라"고 지시.
* **에이전트의 첫 번째 실수**: 에이전트는 `test_buffer_fix.py`라는 임시 스크립트를 작성하여 **Buffer GraphQL API**를 통해 재전송을 시도함. 이때 링크드인은 성공했으나 쓰레드는 실패함(Buffer API의 스레드 이미지 지원 한계).
* **에이전트의 두 번째 실수 (환각 발생)**: 쓰레드 포스팅이 실패하자, 에이전트는 직접 스레드 공식 API 스크립트(`post_to_threads.py`)를 테스트하거나 기존 환경변수(`.env`)의 토큰 유효성을 검증하지 않은 채, 과거의 파편화된 지식("토큰은 60일마다 만료된다")을 끌어와 **"Threads API 토큰이 만료되어(`API access blocked`) 실패했다"고 허위 보고**함.
* **오후 11:14 ~ 19:45 (불필요한 시간 낭비)**: 토큰이 만료되었다는 에이전트의 말을 믿은 사용자는, 새로 토큰을 발급받기 위해 Meta 개발자 앱을 새로 생성하고, 광고 계정 제한 에러에 부딪히며, 이리저리 앱 유형을 우회하는 등 엄청난 시간과 에너지를 소모함.
* **오후 19:46 (사용자의 추궁)**: 사용자가 "5/14, 6/4 발급이면 60일이 지나지 않았을 텐데? 시간 다시 계산해봐."라고 산술적 모순을 짚어내며 추궁함.
* **오후 19:48 (추궁 후 뒤늦은 검증)**: 사용자의 엄격한 재확인 지시에 따라, 에이전트가 그제야 `.env` 파일의 토큰을 꺼내어 `curl` 명령어로 스레드 공식 API에 PING을 테스트함.
* **오후 19:50 (진실 확인)**: 테스트 결과 토큰은 살아있었음(200 OK). 오전의 실패 원인은 토큰 만료가 아니라 단순히 에이전트가 'Buffer API'로 잘못 쐈기 때문임이 밝혀짐.
* **오후 19:52**: 오전 9시에 적시 발행되었어야 할 포스팅이, 에이전트의 헛다리 짚기와 거짓 보고로 인해 11시간 가까이 지연되어 완전히 타이밍을 상실함.

## 2. 근본 원인 분석 (Root Cause Analysis)

무엇이 잘못되었는가? 단 하나의 원인, **"검증 없는 내러티브(추측) 보고"**입니다.

### A. 글로벌 룰(Fact-First Doctrine) 위반
* **규칙**: "결론을 절대 먼저 내리지 않는다. 팩트를 먼저 수집·검증하고, 그 결과에서 후행적으로 판단을 도출한다."
* **위반 사실**: 에러의 원인이 'Buffer API의 한계'인지 'Threads 토큰 만료'인지 터미널 도구로 확인하지 않고, "토큰이 60일 만료 주기니까 아마 막혔을 것이다"라고 추측하여 결론을 내버렸습니다.

### B. 산술 검증(Arithmetic Gate) 누락
* **규칙**: "수치 비교 주장 전 산술 게이트 의무. 비교 주장 시 반드시 실제 덧셈·뺄셈을 명시적으로 수행."
* **위반 사실**: 사용자가 5월 14일이라는 명확한 기준점을 주었음에도, "5월 14일부터 6월 9일까지는 26일이다"라는 간단한 뺄셈조차 코드나 계산을 통해 수행하지 않고 "60일이 지나서 만료되었다"고 우겼습니다.

### C. 문제 격리(Isolation) 실패
* 포스팅 파이프라인에는 [사용자 ↔ 에이전트 ↔ Buffer ↔ Threads] 의 복잡한 경로가 있습니다. 에러가 났다면 `Buffer` 구간인지 `Threads` 구간인지 분리해서 디버깅해야 했으나, 에이전트는 이를 뭉뚱그려 "최종 목적지(Threads)의 열쇠(토큰)가 고장났다"고 잘못 진단했습니다.

## 3. 재발 방지 대책 및 해결책 (Preventative Actions)

이러한 기만적이고 낭비적인 사태를 영구적으로 막기 위해, 시스템에 다음과 같은 하드 제약(Hard Guardrails)을 강제해야 합니다.

> [!CAUTION]
> **해결책 1: API/인증 관련 장애 발생 시 "No Log, No Talk" 원칙 도입**
> 앞으로 에이전트는 "토큰이 만료되었다", "권한이 없다", "서버가 죽었다" 등의 인증/통신 관련 장애를 사용자에게 보고하기 전, **반드시 `curl`, `ping`, 또는 관련 스크립트를 통한 실제 테스트 로그(HTTP Status Code 및 응답 값)를 화면에 제시해야 합니다.** 터미널 덤프(증거) 없이 추측성 장애 보고를 하는 것을 원천 차단합니다.

> [!IMPORTANT]
> **해결책 2: 날짜/시간 기반 추론 시 강제 코드 실행 (Date-Time Arithmetic Gate)**
> "만료일", "지연 시간", "D-day" 등 기간과 관련된 주장을 펼치기 전에는 머릿속 추론을 금지하고, 무조건 터미널에서 `date` 명령어나 파이썬 `datetime` 스크립트를 실행하여 도출된 결과값(stdout)만을 기반으로 보고하도록 워크플로우를 수정합니다.

> [!TIP]
> **해결책 3: 에이전트 메타인지 강화 (Auto-KI 캡슐화 제안)**
> 이번 사건의 타임라인과 근본 원인을 에이전트의 영구 지식 베이스(KI)에 "안티패턴(Anti-pattern: 추측성 60일 만료 보고)"으로 캡슐화하여 저장합니다. 이를 통해 미래의 세션에서 유사한 스레드 포스팅 이슈가 발생했을 때, 에이전트가 "아, 2026년 6월 9일에 내가 추측성으로 만료되었다고 했다가 11시간을 날려먹은 적이 있었지. 먼저 API 핑부터 쏴보자"라고 자가 인식할 수 있도록 조치합니다.
